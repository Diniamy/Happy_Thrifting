BAB I
PENDAHULUAN
1.1Latar Belakang
Seiring dengan berkembangnya era digital ini, perkembangan sistem perangkat lunak dituntut untuk memiliki kualitas yang baik, fleksibel dalam melakukan pengembangan, serta mudah dalam melakukan pemeliharaan. Namun, pada kenyataannya, masih banyak pengembang atau developer perangkat lunak yang meghasilkan kode program yang tidak terstruktur, sulit dibaca, serta menyulitkan proses pengembang lebih lanjut. Hal ini disebabkan oleh kurangnya pemahaman dan penerapan prinsip desain yang tepat dalam proses pengembang perangkat lunak.
Salah satu solusi untuk mengatasi permasalahan tersebut adalah dengan menerapkan prinsip desain SOLID. Prinsip SOLID merupakan seperangkat pedoman dalam pemrograman berorientasi objek. Maka dengan menerapkan prinsip SOLID ini pengembang dapat membangun struktur kode yang bersih, terorganisir, dan mudah untuk diuji. Sehingga mampu meningkatkan efisiensi pemeliharaan serta kualitas perangkat lunak secara keseluruhan.[1]
SOLID Principle pertama kali diperkenalkan oleh Robert C. Martin (Uncle Bob) sebagai panduan desain dalam pemrograman berorientasi objek. SOLID Principle ini memiliki lima prinsip utama yaitu Single Responsibility, Open/Closed, Liskov Subsitution, Interface Segregation, dan  Depedency Inversion bertujuan untuk membuat arsitektur perangkat lunak yang lebih tangguh terhadap perubahan, serta memudahkan kolaborasi dalam melakukan pengembangan. Penerapan prinsip ini terbukti mampu meningkatkan keterbacaan, pengujian, dan pemeliharaan perangkat lunak, terutama pada project bersekala menengah hingga besar.
Prinsip desain SOLID telah banyak digunakan dalam pengembangan perangkat lunak modren untuk menciptakan sistem yang modular dan fleksibel. Pendekatan ini memberikan panduan dalam penyusunan kode yang lebih terstruktur, sehingga memudahkan proses perawatan dan dan pengembangan lebih lanjut. Dengan menerapkan prinsip SOLID, perubahan pada sistem dapat diakomodasi tanpa menimbulkan ketergantungan kompleks antar komponen.[2]
Penelitian ini mengambil studi kasus aplikasi Happy Thrifting, yaitu sebuah platfrom berbasis web yang menjual beli baju bekas secara online. Pada sistem sebelumnya, struktur kode yang diterapkan masih tidak tertata dengan baik, tidak modular, dan sulit untuk dipahami. Oleh karena itu, dalam penelitian ini dilakukan pengembangan ulang terhadap struktur kode dengan menggunakan prinsip desain SOLID agar dapat menghasilkan kode yang bersih, modular, dan mudah untuk diuji, serta meningkatkan keteraturan struktur kode, mempermudah proses pemeliharaan, serta mendukung pengembang sistem secara berkelanjutan.
1.2Hipotesis Awal
Berdasarkan latar belakang yang telah diuraikan, hipotesis awal dalam penelitian ini adalah:
Penerapan prinsip desain SOLID dalam pengembangan aplikasi web Happy Thrifting dapat menghasilkan struktur kode yang lebih modular, terorganisir, mudah diuji, serta meningkatkan kualitas pemeliharaan dan fleksibilitas sistem secara keseluruhan
1.3Rumusan Masalah
Berdasarkan latar belakang di atas, maka rumusan masalah yang akan dibahas dalam penelitian ini adalah:
1.Bagaimana penerapan prinsip SOLID dalam pengambangan aplikasi web Happy Thrifting?
2.Bagaimana struktur sistem yang modular dan terorganisir dapat dibangun mengunakan prinsip-prinsip SOLID?
1.4Tujuan Penelitian
Adapun tujuan dari penelitian ini adalah sebagai berikut:
1.Menerapkan prinsip SOLID dalam pengembangan aplikasi web Happy Thrifting untuk menciptakan sistem yang lebih modular dan maintainable.
2.Merancang struktur kode yang terorganisir dan fleksibel dengan berpedoman pada prinsip SOLID.
3.Mengevaluasi hasil penerapan SOLID dari sisi kemudahan pemeliharaan, keterbacaan kode, dan efisiensi pengembang.
1.5Manfaat Penelitian
Penelitian ini diharapkan memberikan manfaat sebagai berikut:
1.Memberikan referensi praktis bagi pengembang perangkat lunak dalam menerapkan prinsip desain SOLID dalam pengembangan aplikasi web.
2.Menambah wawasan akademik terkait pentingnya struktur kode yang baik dan modular dalam pengembangan perangkat lunak.

Penelitian terdahulu yang dilakukan oleh Manalu Josua (2024) Universitas Multi Data Palembang dengan judul “Metode Clean Architecture Berdasarkan Prinsip Desain Solid Untuk Analisis Aplikasi Pencatatan Absensi Daring (Panda)” Aplikasi PANDA merupakan aplikasi berbasis android dengan bahasa Java yang dimiliki oleh Universitas Multi Data Palembang untuk melakukan absensi dosen dan karyawan. Refactoring dilakukan pada tujuh class yang masih tidak sesuai dengan prinsip Single Responsibility dengan memindahkan function-function yang memiliki tanggung jawab yang sama ke dalam suatu class. Prinsip Interface Segregation diimplementasikan dengan menghapus function implementasi yang tidak diperlukan. Konsep Clean Architecture juga telah diterapkan dengan membuat domain layer yang berisi entities yang telah dimodifikasi agar tidak terpengaruh oleh framework luar dan use case yang berisi business logic aplikasi sehingga presentation layer tidak bergantung secara langsung pada data layer melainkan bergantung kepada domain layer. Penerapan Clean Architecture dan prinsip desain SOLID menghasilkan jumlah source code yang lebih sedikit dan rekomendasi kode dari penelitian ini diharapkan dapat memudahkan developer dalam pemeliharaan dan pengembangan aplikasi PANDA ke depan. 2.2Landasan Teori
Pada poin ini akan menjelaskan teori-teori yang berkaitan dengan pokok permasalahan yang menjadi landasan bagi pemahaman dalam menyusun skripsi ini sebagai berikut:
2.2.1Prinsip SOLID

Gambar 2. 1 SOLID Principle
	Menurut Robert C. Martin, SOLID Principle adalah kumpulan lima prinsip desain berorientasi objek yang bertujuan untuk membuat kode lebih mudah dipahami, dikelola, dan diperluas. Prinsip-prinsip ini adalah: Single Responsibility Principle, Open/Closed Principle, Liskov Substitution Principle, Interface Segregation Principle, dan Dependency Inversion Principle. [16]
Prinsip SOLID merupakan lima prinsip dasar dalam desain perangkat lunak berorientasi objek yang bertujuan untuk meningkatkan kualitas perangkat lunak, khususnya dalam fleksibilitas, keterbacaan, dan kemudahan pemeliharaan. SOLID merupakan akronim dari lima prinsip utama yaitu:[17]

1.Single Responsibility Principle (SRP)
Penjelasan:
Prinsip ini menyatakan bahwa setiap kelas hanya boleh memiliki satu tanggung jawab. Artinya, satu class tidak boleh mencampurkan logika yang berbeda agar kode lebih mudah dipelihara.
Code:
<?php
class RegisterUserService {
    public function execute(array $data) {
        return User::create($data);
    }
}

class SendWelcomeEmail {
    public function execute(User $user) {
        Mail::to($user->email)->send(new WelcomeMail($user));
    }
}

2.Open/Closed Principle (OCP)
DPenjelasan:
Prinsip ini menyatakan bahwa kelas harus terbuka untuk ekstensi, namun tertutup untuk modifikasi. Jadi, jika ada fitur baru, kita cukup menambahkan class baru tanpa mengubah class yang sudah ada.efinisi: kelas terbuka untuk ekstensi, tetapi tertutup untuk modifikasi.
Code:
<?php
interface PaymentMethod { public function pay($amount); }

class CreditCardPayment implements PaymentMethod {
    public function pay($amount) { return "Pay $amount with Credit Card"; }
}

class PaypalPayment implements PaymentMethod {
    public function pay($amount) { return "Pay $amount with PayPal"; }
}

3.Liskov Substitution Principle (LSP)
Penjelasan:
Prinsip ini menyatakan bahwa subclass harus bisa digunakan menggantikan superclass tanpa mengubah perilaku program.
Code:
<?php
abstract class Notification { abstract public function send($msg); }

    public function send($msg) { Mail::raw($msg, fn($m)=>$m->to("user@test.com")); }
}

class SmsNotification extends Notification {
    public function send($msg) { echo "SMS: $msg"; }

4.Interface Segregation Principle (ISP): 
Penjelasan:
Prinsip ini menyarankan agar interface tidak terlalu besar, sehingga class tidak dipaksa mengimplementasikan method yang tidak dibutuhkan.
Code:
<?php
interface Printer { public function print(); }
interface Scanner { public function scan(); }

class AllInOne implements Printer, Scanner {
    public function print() { echo "Printing"; }
    public function scan() { echo "Scanning"; }
}


5.Dependency Inversion Principle (DIP):
Penjelasan:
Prinsip ini menyarankan agar interface tidak terlalu besar, sehingga class tidak dipaksa mengimplementasikan method yang tidak dibutuhkan.
Code:
<?php
interface UserRepository { public function save($data); }

class MySQLUserRepository implements UserRepository {
    public function save($data) { return User::create($data); }

    // Proses login
    public function login(Request $request)
    {
        $credentials = $request->validate([
            'email' => 'required|email',
            'password' => 'required',
        ]);

        // Coba autentikasi pengguna
        if (Auth::attempt($credentials, $request->remember)) {
            $request->session()->regenerate();

            // Periksa role pengguna
            if (Auth::user()->role !== 'user') {
                // Logout jika peran tidak sesuai
                Auth::logout();
                $request->session()->invalidate();
                $request->session()->regenerateToken();

                // Redirect ke login dengan pesan error
                return back()->withErrors([
                    'email' => 'You are not authorized to access this page.',
                ])->withInput();
            }

            // Redirect ke halaman indeks jika peran valid
            return redirect()->intended('/');
        }

        // Jika gagal autentikasi, kembali ke halaman login dengan pesan error
        return back()->withErrors([
            'email' => 'The provided credentials do not match our records.',
        ])->withInput();
    }

    // Logout
    public function logout(Request $request)
    {
        Auth::logout();

        $request->session()->invalidate();
        $request->session()->regenerateToken();

        return redirect('/');
    }

    // Proses register
    public function register(Request $request)
    {
        // Validasi input
        $request->validate([
            'name' => 'required|string|max:255',
            'email' => 'required|string|email|max:255|unique:users',
            'alamat' => 'required|string|max:255',
            'password' => 'required|string|min:8|confirmed',
            'role' => 'user', // Role default
        ]);

        // Buat user baru
        $user = User::create([
            'name' => $request->name,
            'email' => $request->email,
            'alamat' => $request->alamat,
            'password' => Hash::make($request->password), // Hash password
        ]);

        // Login otomatis setelah register

        // Redirect ke halaman utama setelah berhasil register
        return redirect()->route('index')->with('success', 'Account successfully created!');
    }

Pendekatan ini tidak sesuai dengan prinsip Single Responsibility Principle (SRP), karena satu kelas atau komponen menangani berbagai fungsi sekaligus. Selain itu, struktur ini juga menyulitkan proses pengujian dan pemeliharaan karena keterkaitan antar bagian yang sangat erat.

3.4.5.Evaluasi Penerapan Prinsip SOLID
Evaluasi dilakukan untuk mengetahui sejauh mana penerapan prinsip SOLID dapat meningkatkan kualitas kode pada aplikasi Happy Thrifting berbasis web. Evaluasi ini difokuskan pada perbandingan struktur kode sebelum dan sesudah dilakukan refaktorisasi dengan menerapkan prinsip SOLID.
Tujuan dari evaluasi ini adalah untuk melihat apakah terdapat perubahan yang signifikan dalam hal keterbacaan, kemudahan pemeliharaan, dan kerapihan struktur kode setelah prinsip-prinsip SOLID diterapkan. Selain dilakukan secara manual, evaluasi juga dibantu dengan menggunakan tools Laravel Insights yang dapat memberikan penilaian terhadap kualitas kode secara kuantitatif berdasarkan beberapa aspek seperti kompleksitas, arsitektur, dan kepatuhan terhadap prinsip desain. Hasil dari evaluasi ini akan menjadi acuan untuk menilai apakah penerapan prinsip SOLID mampu memberikan dampak positif dalam proses pengembangan dan pengelolaan aplikasi ke depannya.
A.Hasil Evaluasi dengan Laravel Insights
Sebelum menampilkan hasil evaluasi, perlu disampaikan bahwa penggunaan Laravel Insights dalam penelitian ini masih berada pada tahap awal analisis kualitas kode. Dengan kata lain, hasil yang ditampilkan di bagian ini hanya memberikan gambaran awal mengenai kondisi kode sebelum dan sesudah dilakukan refaktorisasi serta penerapan prinsip SOLID. Analisis yang lebih mendalam dapat dilakukan pada tahap selanjutnya agar hasil yang diperoleh menjadi lebih lengkap dan komprehensif.
Tools ini memberi skor berdasarkan beberapa indikator seperti:
1.Code Quality = Mengukur kualitas penulis kode secara keseluruhan.
2.Complexity = Menilai kompleksitas logika program.
3.Architecture = Mengukur Kepatuhan struktur terhadap arsitekture yang baik.
4.Style = Mengukur konsistensi gaya penulisan kode

Sebelum refaktoriasi	  Sesudah refaktorisasi
<? php 
public function store(Request $request)
{
    $product = new Product();
    $product->name = $request->name;
    $product->price = $request->price;

    if ($request->price < 0) {
        return redirect()->back()->withErrors('Harga tidak boleh negatif');
    }

    $product->save();
    return redirect()->route('products.index');
}	<? php 
public function store(ProductRequest $request, ProductService $productService)
{
    $productService->create($request->validated());
    return redirect()->route('products.index');
}

  Penjelasan:
Sebelum refaktorisasi, kode di dalam controller menangani terlalu banyak tanggung jawab sekaligus, yaitu validasi input, logika bisnis, dan penyimpanan data. Setelah refaktorisasi:
a.Validasi dipisahkan ke ProductRequest.
b.Proses penyimpanan produk dipisahkan ke dalam class ProductService.
Dengan cara ini, controller hanya bertanggung jawab untuk menangani request dan response, sesuai prinsip SRP. Sementara itu, class ProductService di-inject menggunakan dependency injection (DIP), sehingga mudah untuk diuji atau diganti jika diperlukan.
